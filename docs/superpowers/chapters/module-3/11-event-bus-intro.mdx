# Event bus intro

Extensions can communicate with each other through pi's shared event bus.

## The pi.events API

```typescript
// Emit an event
pi.events.emit("my:notification", { message: "Hello", from: "extension-a" });

// Listen for an event
pi.events.on("my:notification", (data) => {
  console.log("Received:", data);
});

// Listen once
pi.events.once("my:ready", () => {
  console.log("Extension is ready!");
});

// Unsubscribe
const handler = (data) => { ... };
pi.events.on("my:event", handler);
pi.events.off("my:event", handler);
```

## Why Use the Event Bus?

| Use Case | Example |
|----------|---------|
| **Decoupling** | Extension A notifies Extension B without direct imports |
| **Coordination** | One extension triggers another's behavior |
| **Logging** | Central extension collects events from all others |

## Example: Notification Bridge

Extension A (emitter):
```typescript
pi.registerCommand("notify", {
  handler: async (args, ctx) => {
    pi.events.emit("notification", { message: args });
  },
});
```

Extension B (listener):
```typescript
pi.events.on("notification", (data) => {
  ctx.ui.notify(data.message, "info");
});
```

Now `/notify Hello` in Extension A triggers Extension B!

<Callout variant="tip" title="Event Naming">
Use namespaced events like `my-ext:action`. This avoids collisions between extensions.
</Callout>

## Built-in Events

pi emits its own events you can listen to:

| Event | Data | When |
|-------|------|------|
| `session_start` | `{ cwd, sessionId }` | New session |
| `session_end` | `{ cwd, sessionId }` | Session closed |
| `tool_call` | `{ toolName, input }` | Before tool executes |
| `tool_result` | `{ toolName, output }` | After tool completes |

## Summary

- ✓ `pi.events.emit()` sends events
- ✓ `pi.events.on()` subscribes
- ✓ `pi.events.once()` for one-time
- ✓ `pi.events.off()` to unsubscribe
